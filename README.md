# Taskmesh

Taskmesh is a Discord-centered multi-agent orchestration service for Claude Code, Codex, and Gemini.

The intended deployment target is a single local MacBook running the bot and agent CLIs as background services.

The initial scaffold focuses on:

- Discord channel and thread scoped sessions
- Agent-to-agent delegation through a shared orchestrator
- Shared tools and plugin surface via adapter interfaces
- Storage abstractions for conversation history, task state, and run logs

## Planned runtime

- `Discord Gateway` receives messages, attachments, mentions, and thread events
- `Router` resolves the target agent and creates a task
- `Orchestrator` manages sessions, delegation, and result delivery
- `Agent adapters` wrap Claude Code, Codex, and Gemini
- `Storage` persists channels, sessions, tasks, and events
- `launchd` keeps the Taskmesh service alive on macOS

## Message routing

Taskmesh currently understands these Discord entry forms:

- `@claude fix the failing tests`
- `@codex review this module`
- `@gemini summarize this thread`
- `/claude fix the failing tests`
- `/ask codex review this module`
- `gemini: summarize this thread`

When no explicit target is provided, Taskmesh routes to the configured default agent.

## Discord layout

The current intended Discord structure is:

- category `Mac Book Pro M5 Max`
- channels `#claude-code`, `#codex`, `#gemini`
- category `Agent Orchestra`
- one or more session channels under `Agent Orchestra`

Behavior:

- direct category channels are pinned to a single agent
- `#claude-code` always routes to Claude
- `#codex` always routes to Codex
- `#gemini` always routes to Gemini
- channels under `Agent Orchestra` are treated as shared session workspaces
- in orchestra channels, any agent may delegate to any other agent
- the Discord channel itself is the session boundary for orchestration

## Bot topology

Taskmesh uses four Discord bot identities:

- `Taskmesh Orchestrator Bot`: routing, session notices, final integrated replies
- `Claude Bot`: Claude task start and completion messages
- `Codex Bot`: Codex task start and completion messages
- `Gemini Bot`: Gemini task start and completion messages

The orchestrator bot is the only message receiver. The other three bots are sender identities used to post agent-specific updates into channels.

## Configuration

Copy `.env.example` to `.env` and set:

- Discord bot credentials
- storage driver and database path
- optional CLI command and args for each agent

If an agent command is not configured, Taskmesh falls back to a stub adapter for that agent.

## Local background runtime

Taskmesh is being shaped for local execution on this MacBook:

- the Discord bot runs as a background Node service
- Claude Code, Codex, and Gemini are invoked from locally installed CLIs
- persistent state lives in a local SQLite database
- `launchd` is the preferred supervisor on macOS

Add the four bot tokens to `.env` before starting the service.

For local file-writing tasks, the default CLI examples assume the agents are allowed to operate inside `/Users/sumin/taskmesh`.
Those example args currently include dangerous permission-bypass flags for local unattended use on this MacBook.

## Delegation protocol

Taskmesh now supports a simple output-driven delegation protocol for local agent CLIs.

- an agent can emit `TASKMESH_DELEGATE codex :: review the patch`
- Taskmesh turns that into a child task
- the child result is appended back into the parent task summary

The current stub adapter also supports a local test format:

- `[delegate:codex] review this patch`

## File creation protocol

Agents can ask Taskmesh to create files and upload them back to Discord.

Use this format in agent output:

```text
TASKMESH_WRITE_FILE notes/summary.md
<<<TASKMESH_CONTENT
# Summary
Generated by the agent.
TASKMESH_END_CONTENT
```

Taskmesh will:

- write the file under the configured artifacts directory
- keep the generated file on disk
- upload the file back to Discord as an attachment
